# 双目标定与深度测距说明

这份文档记录当前软件的双目标定、摄像头稳定性、深度估计和界面显示逻辑。后续遇到类似问题时，优先对照这里排查。

## 当前硬件与标定板

当前使用两只独立 USB 摄像头，不是硬件同步双目模组。两路图像通过软件分别采集，再用于标定和深度估计。

当前 AprilTag 标定板信息：

- Dictionary: `DICT_APRILTAG_36h11`
- 板面文字：`5 x 7`
- 软件中实际应填写：`markers_x = 7`，`markers_y = 5`
- `board_mirror_x = true`
- Tag size: `80 mm`，即 `0.08 m`
- Tag spacing: `24 mm`，即 `0.024 m`

注意：不能直接按板面文字把 OpenCV 参数填成 `5 x 7`。这块板在当前拍摄图像中对应的是 `7 x 5` 且 X 方向镜像。如果填错，软件仍可能检测到所有 tag，但三维物理点会对应到错误位置，导致重投影误差和 stereo error 异常巨大。

AprilTag 36h11 检测时使用 `markerBorderBits = 2`。如果这个值不对，可能出现板子很清楚但检测不到 tag 的问题。

## 标定的目的

双目标定不是为了直接得到深度图，而是为了得到两台相机的几何参数：

- `K1`, `D1`: 左相机内参和畸变
- `K2`, `D2`: 右相机内参和畸变
- `R`, `T`: 右相机相对左相机的旋转和平移
- `R1`, `R2`: 双目矫正旋转
- `P1`, `P2`: 矫正后的投影矩阵
- `Q`: 视差转三维坐标的重投影矩阵

当前代码流程：

1. `aruco_board.py` 检测左右图中的 AprilTag。
2. `common_board_points()` 找出左右都看到的相同 tag 角点。
3. `calibration.py` 中 `collect_calibration_points()` 生成 `obj_points`、`left_points`、`right_points`。
4. `cv2.calibrateCamera()` 分别求左右相机内参。
5. `cv2.stereoCalibrate()` 求左右相机之间的 `R`、`T`。
6. `cv2.stereoRectify()` 生成 `R1`、`R2`、`P1`、`P2`、`Q`。
7. 标定结果保存到 `data/output/stereo_calib.npz` 和 `data/output/stereo_calib.yaml`。

## 单摄标定

单摄模式现在也支持标定，但它和双目标定完全分开：

- 单摄图片保存到 `data/images/single/`。
- 单摄标定输出保存为 `data/output/mono_calib.npz` 和 `data/output/mono_calib.yaml`。
- 双摄图片仍保存到 `data/images/left/` 和 `data/images/right/`。
- 双目标定输出仍保存为 `data/output/stereo_calib.npz` 和 `data/output/stereo_calib.yaml`。

单摄标定只会得到：

- `K`: 单个相机内参
- `D`: 单个相机畸变
- `error`: 单目标定重投影误差
- `image_size`: 标定图像尺寸

单摄标定不会得到双目基线 `T`，也不会生成 `Q` 矩阵，所以不能用于 `Depth / Distance` 双目测距。它适合用于：

- 获取单个相机内参。
- 检查镜头畸变。
- 给单相机图像做去畸变。
- 在做双目标定前，单独确认某个摄像头和标定板参数是否正常。

使用方式：

1. 勾选 `Single camera mode`。
2. 打开摄像头。
3. 点击 `Capture Pair`，此时会保存单摄图片到 `single/` 目录。
4. 拍摄多张不同姿态、不同位置、不同距离的标定板图片。
5. 点击 `Start Calibration`，软件会执行单目标定并输出 `mono_calib.*`。

注意：单摄标定完成后，软件不会创建 `Rectifier`，也不会启用双目深度计算。需要深度测距时，必须退出单摄模式，采集左右图像对并生成 `stereo_calib.*`。

## 2D 平面测距

`2D Measurement` 模式用于单相机平面测距，不使用双目的视差深度。它依赖单目标定输出：

- 标定文件：`data/output/mono_calib.npz`
- 使用参数：`K` 和 `D`
- 手动输入：`Plane distance mm`

`Plane distance mm` 表示相机到被测平面的垂直工作距离，单位是 `mm`。如果下一次测试时相机高度变了，只要相机本身、镜头焦距、分辨率没有变化，不需要重新做内参标定，但必须把这个工作距离改成新的真实距离。

当前实现假设被测平面与相机成像平面平行，暂时不做倾斜角补偿。如果相机相对被测平面有明显倾斜，测出来的 2D 距离会有系统误差。

软件计算流程：

1. 在 `2D Measurement` 模式点击 `Load Mono Calibration`，加载 `mono_calib.npz`。
2. 在 `Plane distance mm` 输入相机到被测平面的垂直距离。
3. 在预览图或原图窗口中点击两个点。
4. 软件先用 `cv2.undistortPoints()` 根据 `K`、`D` 去畸变，并得到归一化相机坐标。
5. 用输入的工作距离 `Z` 把归一化坐标换算到平面上的毫米坐标。
6. 计算两点的欧氏距离，底部显示 `2D: xx.xx mm`。

核心关系可以理解为：

```text
X = x_norm * Z
Y = y_norm * Z
distance = sqrt((X2 - X1)^2 + (Y2 - Y1)^2)
```

其中 `x_norm`、`y_norm` 是去畸变后的归一化坐标，`Z` 就是界面里输入的 `Plane distance mm`。

这个功能适合测量同一个平面上的孔距、边长、间距等二维尺寸。不要把它当成双目深度测距使用；它不能自动知道物体高度，也不能在不同深度的多个平面之间自动测距。

## 条码识别

`Barcode Detection` 模式用于从当前摄像头画面中解码条码。当前软件使用 `zxing-cpp`，支持：

- `Code 39`
- `Code 128`
- `Codabar`
- `EAN`
- `ITF25`
- `Code 93`
- `QR Code`
- `DataMatrix`

使用方式：

1. 打开摄像头。
2. View 切换到 `Barcode Detection`。
3. `Type` 选择预期条码类型；不知道类型时可以用 `All`。
4. `Confirm frames` 设置连续确认帧数，默认 `3`。
5. 条码连续识别到相同内容后，底部显示 `Barcode: xxx`，状态栏打印 `Barcode confirmed`。

稳定性逻辑：

- 解码逻辑在 `barcode.py` 中独立实现，输入是 OpenCV 图像帧 `np.ndarray`。后面接网络工业相机时，只要能拿到图像帧，就可以复用这套解码逻辑。
- 同一个条码内容和格式连续出现多帧后才确认，减少单帧误识别。
- 预览图会用绿色框标出识别到的条码位置。
- 如果没有安装 `zxing-cpp`，软件不会崩溃，而是在界面提示重新执行 `python -m pip install -r requirements.txt`。

为了提高稳定解码率，优先保证：

- 条码清晰对焦。
- 不要过曝，白底不能一片死白。
- 一维码最细条宽在图像中至少有数个像素。
- QR Code 和 DataMatrix 的最小模块在图像中至少有数个像素。
- 尽量固定曝光、增益和光源。
- 现场使用时优先选择具体条码类型，少用 `All`，这样更快，也能减少误识别。

注意：软件解码可以先支持当前 USB 摄像头调试，但长期工业现场建议使用定焦工业相机、合适镜头、稳定光源和固定安装结构。后续接入网络工业摄像头时，应把相机取流作为新的帧来源，条码解码模块不需要重写。

## 标定图片质量判断

软件现在把每组图片分成三类：

- `VALID`: 至少 `40` 个共同角点，参与最终标定。
- `WEAK`: `16-39` 个共同角点，可预览分析，但不参与最终标定。
- `INVALID`: 少于 `16` 个共同角点，不参与最终标定。

标定时会在预览图上显示每张图的检测结果。标定完成后，可以通过顶部的 `Calibration Pair` 下拉框或 `Prev Pair` / `Next Pair` 逐张查看效果。

当前这批图像在正确板模型下，曾得到：

```text
left error: 1.5237
right error: 1.5380
stereo error: 1.5336
baseline: 0.0647 m
quality: CHECK
```

`CHECK` 代表可以用于功能验证和粗略测距，但不是高精度状态。若要提升精度，目标是把 stereo error 尽量压到 `1 px` 以下。

## 深度估计原理

深度估计基于视差：

```text
Z = f * B / d
```

含义：

- `Z`: 深度，单位 m
- `f`: 矫正后的焦距，单位 px
- `B`: 双目基线，单位 m
- `d`: 视差，单位 px

当前标定结果大致为：

```text
f ≈ 519.7 px
B ≈ 0.0647 m
f * B ≈ 33.63
```

所以可以粗略估算：

```text
Z ≈ 33.63 / d
```

例如：

```text
d = 30 px  -> Z ≈ 1.12 m
d = 50 px  -> Z ≈ 0.67 m
d = 80 px  -> Z ≈ 0.42 m
d = 120 px -> Z ≈ 0.28 m
```

代码中不是手写这个公式，而是在 `rectification.py` 里用：

```python
cv2.reprojectImageTo3D(disparity, Q)
```

`Q` 矩阵已经包含了焦距、基线和主点信息。

## xL、xR 与视差

如果要手算某个点的深度，需要知道：

```text
d = xL - xR
```

- `xL`: 该点在左侧矫正图里的横坐标。
- `xR`: 同一个真实点在右侧矫正图里的横坐标。
- `d`: 视差。

双目矫正后，同一个真实点理论上应位于左右图同一水平线。因此手动找点时，只需要在右图同一行寻找对应特征点。

程序实际使用 `StereoSGBM` 自动计算每个像素的视差。若程序得到：

```text
xL = 321
disparity = 35.1
```

则：

```text
xR = 321 - 35.1 = 285.9
Z ≈ 33.63 / 35.1 = 0.958 m
```

## 深度图为什么会有很多无效区域

当前算法是 OpenCV `StereoSGBM`。它是传统双目匹配算法，需要在左右图中找到相同的小图像块。以下情况会导致无效深度：

- 纯色区域纹理不足。
- 过曝区域变成一片白，细节丢失。
- 反光、透明、金属表面左右图高光位置不同。
- 物体太近，左右遮挡差异变大。
- 真实视差超过 `numDisparities` 搜索范围。
- 左侧天然搜索边界区域无效。
- 左右摄像头不同步，目标或相机运动。
- 标定误差过大，矫正后同一点不在同一水平线上。

成像清晰不等于深度可匹配。SGBM 更依赖纹理、曝光一致性和左右图相似度。

## 深度参数逻辑

`Depth / Distance` 模式下，侧边栏会隐藏标定板参数，显示深度参数：

- `Preset`: `Balanced`、`Near`、`Far / Fast`
- `Display`: `Raw`、`Filtered`
- `numDisparities`
- `blockSize`
- `uniquenessRatio`
- `speckleWindowSize`
- `speckleRange`
- `disp12MaxDiff`

这些参数是动态生效的，不需要重新打开摄像头或重新加载标定。下一帧深度计算会直接使用最新参数。

### numDisparities

`numDisparities` 是最大视差搜索范围，必须是 `16` 的倍数。它相当于“近距离能力”旋钮：

- 值越大，能测更近。
- 值越大，计算越慢。
- 值越大，左侧天然无效区域越宽。

大致关系：

```text
128 -> 适合 30 cm 以上
192 -> 适合 20 cm 以上
256 -> 更近，但左侧损失明显
```

### blockSize

`blockSize` 是匹配块大小：

- 小值细节更多，但噪声更大。
- 大值更稳定，但边缘更糊。

推荐先用 `7`。如果深度图太碎，可试 `9`。

### uniquenessRatio

`uniquenessRatio` 越大，算法越严格：

- `5`: 深度区域更多，但误匹配风险更高。
- `10`: 平衡。
- `15`: 更严格，空洞更多。

### speckle 与一致性过滤

调试时可以放松：

```text
speckleWindowSize = 0
disp12MaxDiff = -1
```

如果这样深度区域明显变多，说明之前是过滤器删掉了部分结果。正式测距时再逐步加回过滤。

## 深度显示与测距分离

当前软件刻意把“显示”和“测距”分开：

- `Raw`: 显示原始 SGBM 视差预览。
- `Filtered`: 只对预览图做小空洞填补和双边滤波，让画面更连续。
- 点击测距仍然使用原始 raw disparity，不使用滤波补出来的假深度。

点击测距现在不是只取单个像素，而是取点击点附近 `5 x 5` 区域内有效视差的中位数。这样可以减少单像素噪声。如果中心点无效，但周围有效，也能得到更稳定的距离。

## ROI 显示逻辑

`numDisparities` 会导致左侧一段区域天然无法完整搜索。为了避免深度图左边一大块黑色区域看起来像画面异常，当前界面做了如下处理：

- 左侧深度图裁掉左边无效区域，让深度预览更居中。
- 右侧矫正图不裁剪，保留完整画面。
- 右侧用绿色框标出当前参数下可计算深度的有效区域。
- 左侧深度图虽然被裁剪，但点击坐标会自动补回偏移，不会影响测距。

## 摄像头稳定性逻辑

当前双摄采集使用左右两个独立线程。每个线程直接 `read()`，读到多少帧就保存多少帧为最新帧。UI 只显示最新帧，不在采集线程里节流。

这样做是因为之前遇到：

- 双摄运行一段时间后越来越卡。
- 某一路摄像头帧更新明显变慢。
- `grab/retrieve` 加 FPS 节流在部分 Windows 后端下导致阻塞。
- 高分辨率下 `isOpened()` 成功但没有稳定视频流。

当前打开逻辑要求 10 秒内持续拿到稳定帧，否则认为打开失败并释放资源，按钮恢复为 `Open Cameras`。打开期间禁用按钮，避免线程未退出时反复点击导致闪退。

更多摄像头排障见 `docs/camera-troubleshooting.md`。

## 提升精度的优先级

如果希望测距更准，优先顺序如下：

1. 提升标定质量，尽量让 stereo error 小于 `1 px`。
2. 固定曝光、增益、白平衡，避免过曝。
3. 增加目标表面纹理，避免纯色和反光区域。
4. 保证左右摄像头稳定输出，不掉帧、不阻塞。
5. 尽量使用稳定的更高分辨率。
6. 如果测远距离，增大双目基线。
7. 如果有条件，使用硬件同步双目、全局快门相机。

更高级的摄像头可以提升稳定性和精度，但不能解决所有问题。纯色、过曝、反光、遮挡区域，即使使用更好的摄像头，也可能没有可靠深度。
